## What Is a Library? What Is a Module?

These words are used interchangeably in conversation. Here is the precise meaning of each:

| Term | Meaning | Example |
|---|---|---|
| **Module** | A single `.py` file you can import | `random` is a file called `random.py` inside Python |
| **Package** | A folder of modules | `pygame` is a folder containing many `.py` files |
| **Library** | General word for importable code — module or package | `random`, `math`, `os` are all "libraries" |
| **Standard library** | Ships with Python — no installation needed | `random`, `math`, `os`, `json`, `time`, `csv` |
| **Third-party library** | Must install with `pip` | `pygame` — you ran `pip install pygame` for this |
| **Custom module** | A `.py` file *you* write that other files import | `support.py`, `save_load.py`, `settings.py` |

**You have already been writing custom modules.** Every `.py` file you created and imported somewhere else is a module. `settings.py` is a module. `save_load.py` is a module. This unit makes that explicit.

## How `import` Works — Three Styles

```python
# Style 1 — import the whole module
# You access everything with the module name as a prefix
import random
random.choice(["a", "b", "c"])     # must write "random." every time

# Style 2 — import specific names from a module
# No prefix needed, but you must know what you are bringing in
from random import choice, randint
choice(["a", "b", "c"])            # works without prefix

# Style 3 — import everything (use sparingly)
from random import *               # dumps ALL names into your file
choice(["a", "b", "c"])            # works, but messy in large projects
```

**Which style should you use?**

Use Style 1 (`import random`) for standard libraries — the prefix makes it obvious where the function comes from when someone reads your code six months later.

Use Style 2 (`from x import y`) when you only need one or two specific things and you will use them constantly.

Avoid Style 3 except in `settings.py` where you intentionally want all constants available.

**Building `drop_loot()` step by step:**

```python
def drop_loot(enemy_level=1):
    # Step 1: Make one roll
    roll = random.random()

    # Step 2: Calculate thresholds that scale with enemy_level
    # Each level adds 1% to legendary and 2% to rare
    legendary_chance = 0.05 + (enemy_level * 0.01)
    rare_chance      = 0.15 + (enemy_level * 0.02)

    # Step 3: Compare roll to thresholds (check most rare first)
    if roll < legendary_chance:
        tier = "legendary"
    elif roll < legendary_chance + rare_chance:
        tier = "rare"
    elif roll < legendary_chance + rare_chance + 0.40:
        tier = "common"
    else:
        return None     # no drop at all

    # Step 4: Filter LOOT_TABLE to only items of the right rarity
    # Then pick one randomly
    from settings import LOOT_TABLE
    options = [item for item in LOOT_TABLE if item["rarity"] == tier]
    return random.choice(options) if options else None
```

**Block 1 — Distance between two points:**

```python
# sandbox_math.py  — Block 1
import math

# The distance between (x1,y1) and (x2,y2) is the hypotenuse of a right triangle
# Pythagorean theorem: distance = sqrt((x2-x1)² + (y2-y1)²)
# math.hypot() does this calculation for you in one call

player_x, player_y = 400, 300
enemy_x,  enemy_y  = 100, 150

dx   = player_x - enemy_x    # 300
dy   = player_y - enemy_y    # 150
dist = math.hypot(dx, dy)    # sqrt(300² + 150²) = sqrt(112500) ≈ 335.4

print(f"Distance: {dist:.1f} px")

# Is the enemy within detection range?
NOTICE_RADIUS = 200
if dist < NOTICE_RADIUS:
    print("Enemy can see the player")
else:
    print("Player is too far away")
```

**Block 2 — What is a Vector2?**

```python
# sandbox_math.py  — Block 2
import pygame
pygame.init()

# A Vector2 is just an (x, y) pair with maths built in
# It supports +, -, *, / and has useful methods

player_pos = pygame.math.Vector2(400, 300)
enemy_pos  = pygame.math.Vector2(100, 150)

# Subtraction gives you a vector POINTING from enemy toward player
diff = player_pos - enemy_pos
print(f"Direction: {diff}")           # Vector2(300, 150)
print(f"Distance:  {diff.length():.1f}")  # 335.4 — same as math.hypot

# .normalize() shrinks the vector to length 1 while keeping direction
# This is called a "unit vector"
unit = diff.normalize()
print(f"Unit vector: ({unit.x:.3f}, {unit.y:.3f})")
print(f"Unit length: {unit.length():.3f}")    # always exactly 1.0

# Move the enemy 3 pixels toward the player
speed   = 3
new_pos = enemy_pos + unit * speed
print(f"New enemy pos: ({new_pos.x:.1f}, {new_pos.y:.1f})")
```

**Block 3 — Why normalise? The diagonal speed problem:**

```python
# sandbox_math.py  — Block 3
import pygame
pygame.init()

# If a player presses RIGHT (1, 0) they move 1 unit per frame
# If they press DOWN  (0, 1) they move 1 unit per frame
# If they press BOTH  (1, 1) — how far do they move?

cardinal  = pygame.math.Vector2(1, 0)   # right only
diagonal  = pygame.math.Vector2(1, 1)   # right AND down

print(f"Cardinal length:  {cardinal.length():.3f}")   # 1.000 — correct
print(f"Diagonal length:  {diagonal.length():.3f}")   # 1.414 — 41% faster!

# Fix: normalise BEFORE multiplying by speed
diagonal_fixed = diagonal.normalize()
print(f"Normalised length: {diagonal_fixed.length():.3f}")   # 1.000 — correct

# In your game code, always do this before moving:
direction = pygame.math.Vector2(1, 1)
if direction.length() > 0:               # guard against (0,0) — can't normalise zero
    direction = direction.normalize()
# Now direction.length() == 1.0 regardless of which keys are held
```

**The spritesheet functions — the most important new addition:**

A **spritesheet** is one image file that contains many frames in a grid. Instead of loading 8 separate PNG files for a walk cycle, you load one file and cut it apart.

**Why `os.path.join()` instead of `path + "/" + name`?**

On Mac and Linux, folders are separated by `/`. On Windows, they are separated by `\`. If you hard-code `/` and a student opens your game on Windows, paths will fail. `os.path.join("graphics", "player", "down")` produces the right separator for whatever OS is running.

### 🧠 Understand It First

**How animation works — the float counter:**

The trick is a floating-point index that increments slowly each frame:

```python
self.frame_index     = 0.0
self.animation_speed = 0.15   # how many frames to advance per game tick

# Every frame in update():
self.frame_index += self.animation_speed

# After 7 ticks: 0.0, 0.15, 0.30, 0.45, 0.60, 0.75, 0.90, 1.05 → wraps to 0.05
# So at 60 FPS: frame changes 60 × 0.15 = 9 times per second

if self.frame_index >= len(frames):
    self.frame_index = 0.0           # wrap back to the start

self.image = frames[int(self.frame_index)]  # int() chops the decimal
```

**Why a float and not an integer counter?**

If you incremented by 1 every frame at 60 FPS, a 4-frame cycle would complete in 4/60 = 0.067 seconds. Characters would blur. With 0.15 per frame, the cycle takes 4/9 ≈ 0.44 seconds — smooth and natural.

**The four-directional status system:**

```python
# status is a string that says which direction the player is facing

    "right": [frame0, frame1, frame2, frame3],

    # Row order in our test sheet: down=0, left=1, right=2, up=3
    frames_per_direction = 4

        self.animations[direction] = all_frames[start:end]
```

**Why `rect.center = centre` after setting a new image:**

Different animation frames might have slightly different bounding boxes. If you just swap `self.image` without updating the rect, the sprite could drift. The fix:

```python
def _animate(self):
    frames = self.animations[self.status]
    self.frame_index += self.animation_speed
    if self.frame_index >= len(frames):
        self.frame_index = 0.0
    self.image = frames[int(self.frame_index)]

    # Keep the player visually centred at the same point after the image changes
    centre          = self.rect.center      # remember where we were
    self.rect       = self.image.get_rect() # new rect from new image
    self.rect.center = centre               # put the new rect at the same centre
```

## Checkpoint 5 — `pygame.time`: Cooldowns and Timed Buffs ⏱️

### 📋 Your Task

Games need time — not just for animation, but for cooldowns (stop button spam), timed effects (a speed boost that wears off), and autosave. This checkpoint adds all three using `pygame.time.get_ticks()`, and wires `draw_bar()` from `support.py` into the HUD so players can see the cooldown recharging.

### 🧠 Understand It First

**`pygame.time.get_ticks()` — the in-game clock:**

```python
import pygame
pygame.init()

# get_ticks() returns the number of milliseconds since pygame.init() was called
# It increases every frame — roughly 16ms per frame at 60 FPS

now = pygame.time.get_ticks()
print(now)   # something like 1247  (ms since the game started)
```

**The cooldown pattern — used in almost every game:**

```python
# Store WHEN the last action happened
self.last_attack_time = 0          # 0 = never attacked
self.attack_cooldown  = 500        # 500ms = half a second between attacks

# When the player tries to attack:
now = pygame.time.get_ticks()
time_since_last = now - self.last_attack_time   # how long ago was the last attack?

if time_since_last >= self.attack_cooldown:
    # Enough time has passed — allow it
    self.last_attack_time = now    # record this as the new "last attack"
    self._do_attack()
else:
    # Not enough time — ignore the input
    # Optionally show how much time is left:
    remaining = self.attack_cooldown - time_since_last
    print(f"Wait {remaining}ms")
```

**Why store a timestamp instead of a countdown?**

Imagine you stored `self.cooldown_remaining = 500` and subtracted from it each frame. At 60 FPS each frame is 16ms, so after 30 frames you have subtracted 480ms — but frames are not always exactly 16ms. If the game stutters, your countdown is wrong.

`get_ticks()` reads the real clock every time, so `now - self.last_attack_time` is always accurate regardless of frame rate wobble.

**Measuring the charge-up fraction for a HUD bar:**

```python
def cooldown_fraction(self):
    """Returns 0.0 (just fired) → 1.0 (fully recharged)"""
    elapsed = pygame.time.get_ticks() - self.last_attack_time
    return min(elapsed / self.attack_cooldown, 1.0)
    # min() prevents going over 1.0 after fully recharged
```

**Timed buffs — the expiry timestamp pattern:**

```python
# Store WHEN the buff expires (not how long is left)
self.active_buffs = {}   # {"speed_boost": 1247000}  ← expiry timestamp in ms

def apply_buff(self, name, duration_ms):
    expiry = pygame.time.get_ticks() + duration_ms   # current time + duration
    self.active_buffs[name] = expiry
    # Apply the effect immediately
    if name == "speed_boost":
        self.stats["Speed"] *= 2

def update_buffs(self):
    now     = pygame.time.get_ticks()
    expired = [name for name, expiry in self.active_buffs.items() if now >= expiry]
    for name in expired:
        del self.active_buffs[name]
        # Reverse the effect
        if name == "speed_boost":
            self.stats["Speed"] //= 2
```
